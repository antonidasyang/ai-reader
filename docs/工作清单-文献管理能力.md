# ai-reader · 文献管理能力 · 工作清单(修订版 v2 · 云端协作方向)

> 本版**取代 v1**,是 `需求-文献管理能力.md`(云端协作方向)的**实现路线**(只讲怎么做;要做什么/验收看需求文档)。
> 定向依据:用户拍板「纯自建后端,PostgreSQL + TypeScript 服务端;AI 数据也要上云共享;引文砍掉;标注/笔记挂起(待体验 Zotero 后定)」。

---

## 0. 本期目标与范围(已修订)

**做(In scope)**
1. 强化文献库管理:结构化题录、组织、检索(在现有 ai-reader 基础上)。
2. **以「研究课题」为单元维护文献库**:每个课题 = 一个独立库。
3. **云端同步**:同一用户多台机器、同一课题多个成员,同步同一个库。
4. **多人协作**:课题成员共享一个库(含成员/权限)。
5. **导入即按标识符自动补全题录**(DOI / arXiv 等)。
6. **AI 数据上云共享**:摘要 / 翻译 / 对话(及将来的标注)也同步、协作者可见。
7. **离线优先**:断网仍能浏览 / 阅读 / 编辑,联网后台同步(保住 ai-reader 的离线强项)。

**不做 / 挂起**
- ❌ **引文 / 参考文献生成**:砍掉(ai-reader 是 reader 不是 paper,引文属写作侧)。
- ⏸️ **标注 / 笔记(comment)**:挂起,等用过 Zotero 再定;**数据模型预留**,不挡路。

**设计原则:离线优先(与需求文档 §2 对齐)**
客户端始终读写**本地**、后台与服务器对账同步,确保「浏览/阅读/编辑/本地检索」**断网可用、联网自动同步**。这是同时满足「云端协作」与「断网核心可用」的关键,贯穿全部 Phase。

---

## 1. 技术选型

**已定**
- 后端:**TypeScript + PostgreSQL**,**纯自建**(不走 BaaS)。
- 后端独立成一个服务/仓库(如 `ai-reader-server`),与 Qt 客户端仓库分开。
- 客户端:沿用 ai-reader 现有 **Qt6 / C++ / QML**,本地用 **SQLite** 做离线优先缓存。

**推荐默认(可改,见 §8 决策点)**
| 层 | 默认选择 | 备选 |
|---|---|---|
| Node 运行时 | Node.js LTS | Bun / Deno |
| 后端框架 | **NestJS**(模块化、依赖注入、Guard 做鉴权/权限,适合多资源 + auth 的正经 API) | Fastify / Hono(更轻) |
| ORM + 迁移 | **Prisma**(类型安全、迁移好用) | Drizzle(SQL-first 更轻)/ Kysely |
| 认证 | 邮箱密码 **argon2** 哈希 + **JWT**(短效 access + 轮换 refresh),`jose` | 加 OAuth(Google)后续 |
| 权限 | 课题成员校验(owner/editor/viewer),写在 Guard/中间件(我们手搓 Supabase 的 RLS 等价物) | — |
| 文件(PDF)存储 | **S3 兼容对象存储**,自托管 **MinIO**;按内容寻址(paperId 去重) | Cloudflare R2 / AWS S3 |
| 实时推送 | **WebSocket** 按课题广播「有变更,拉取 since v」 | SSE / 轮询起步 |
| 同步模型 | **版本号乐观同步(Zotero 风格)** + 字段级后写胜;**离线优先 + 本地 outbox** | 笔记将来可上 CRDT |
| 部署 | **Docker Compose**(PG + MinIO + 服务 + 反代/TLS) | k8s(规模大了再说) |
| 客户端同步层 | C++ `SyncEngine`,复用 `QNetworkAccessManager` + `QWebSocket` | — |

## 2. 总体架构

```
┌──────────────────────── 云端(自建,Docker Compose) ────────────────────────┐
│  ai-reader-server (TS / NestJS)                                              │
│    REST + WebSocket  ──  鉴权/权限  ──  同步引擎(版本号)                     │
│         │                                   │                               │
│   PostgreSQL(题录/课题/成员/标签/AI数据 + 每对象 version)      MinIO(PDF, 按 paperId 去重) │
└──────────────────────────────────────────────────────────────────────────┘
                     ▲  HTTPS / WSS(离线优先,后台同步)
                     │
┌──────────────────── 桌面客户端 ai-reader(Qt/C++/QML,离线优先) ──────────────┐
│  SyncEngine(pull/push/outbox/冲突)  ──  本地 SQLite(库镜像 + FTS5 检索)      │
│  现有:PaperController / BlockCache / ChatService …  ── 经 paperId 桥接复用    │
└──────────────────────────────────────────────────────────────────────────┘
```

**关键桥接(沿用 v1 决策)**:客户端一篇文献的 attachment 记 **`paperId`(首 4MB+大小 的 SHA-256,已存在)**,打开即命中现有 `cache/{blocks,translations,summary,chat}/<paperId>.json`。AI 数据上云时也以 `paperId` 为键,**本地缓存与云端共享数据天然对齐**。

**离线优先如何兼顾「断网可用」**:客户端永远读写**本地 SQLite**;`SyncEngine` 在后台与服务器对账(拉变更、推 outbox、解决冲突)。断网时照常浏览/阅读/编辑,联网自动同步——这正好满足原需求「核心操作断网也能用」。

## 3. 数据模型(PG 为权威,客户端 SQLite 镜像同结构)

每个可同步对象都带:`id UUID`(全局稳定,**不再用自增主键**)、`project_id`、`version BIGINT`(服务器分配、单调递增,作同步游标)、`deleted BOOL`(软删,同步需要)、`updated_at`、`updated_by`。

```sql
users(id uuid pk, email unique, password_hash, display_name, created_at)
projects(id uuid pk, name, description, owner_id, created_at, version)            -- 课题
project_members(project_id, user_id, role, added_at, primary key(project_id,user_id))
                                                       -- role: owner | editor | viewer
items(id uuid pk, project_id, item_type, title, creators jsonb, year int,
      publication, doi, arxiv_id, abstract, csljson jsonb,
      version bigint, deleted bool, created_by, updated_at, updated_by)
attachments(id uuid pk, item_id, paper_id text, storage_key, content_type,
            byte_size bigint, sha256, version bigint, deleted bool)
collections(id uuid pk, project_id, name, parent_id, version bigint, deleted bool)
collection_items(collection_id, item_id, primary key(collection_id,item_id))
tags(id uuid pk, project_id, name)            item_tags(item_id, tag_id)
ai_artifacts(id uuid pk, project_id, paper_id text, type, author_id, payload jsonb,
             model, version bigint, deleted bool, updated_at)   -- type: summary|translation|chat
-- 预留(标注/笔记挂起,先建表不接 UI):
annotations(id uuid pk, item_id, paper_id, author_id, type int, page_index int,
            rects jsonb, color, quoted_text, comment, sort_index, version bigint, deleted bool)
notes(id uuid pk, item_id, author_id, body_md, version bigint, deleted bool, updated_at)
```
**同步游标**:每课题一个 `version` 序列;对象变更即领取新 `version`。客户端按课题存 `lastSyncVersion`,增量拉 `since`。

## Phase 2 · 题录库 + 同步引擎 + 文件存储(本期核心)

**目标**:结构化题录双端落地;增量同步 + 冲突解决;PDF 上云去重;库列表 UI。

数据 + 后端:
- [ ] 2.1 `items` / `attachments` 双端 schema(§3);PG 触发器/服务在写时领 `version`。(M)
- [ ] 2.2 **同步 API(对象通用,不只 items)**:`GET /projects/:id/sync?since=V` 返回该课题自 V 起所有变更对象 + `newVersion`;`POST /projects/:id/push` 批量 upsert,带各对象的 `expectedVersion` → 冲突回 409 + 当前值。(L)
- [ ] 2.3 文件存储:MinIO;`POST /attachments/:id/upload-url`→presigned PUT,`GET …/download-url`;**按 `paperId`/sha256 去重**(同一 PDF 多课题/多人只存一份)。(M)
- [ ] 2.4 WebSocket:`/projects/:id/stream` 广播「changed, version≥V」,触发客户端拉取。(M)

客户端同步引擎(C++):
- [ ] 2.5 `SyncEngine`:① pull(按 `lastSyncVersion` 增量,写本地 SQLite)② push(本地 **outbox** 队列里的变更,带 expectedVersion)③ 冲突:409 → **字段级后写胜**(按 `updated_at`)合并后重推 ④ 离线累积、联网续传。(L · 难点)
- [ ] 2.6 PDF 同步:本地导入即算 paperId,缺则上传、有则秒传;打开课题文献按需下载。(M)
- [ ] 2.7 `LibraryModel` + `qml/LibraryPane.qml`:课题内题录列表(题名/作者/年/出处/类型,排序),点击打开;题录编辑 `MetadataDialog.qml`。接入现有 pane 体系(`LayoutSettings`)。(L)

**判据**:A 机加文献 → B 机/同课题他人秒见;断网编辑、联网自动同步;同 PDF 不重复存。

**抄/砍 Zotero**:抄它的**版本号增量同步 + 软删 + 按内容存储**思路;砍其多库类型/EAV/`*Combined`/翻译沙箱。

## Phase 3 · 导入即自动补全题录

**目标**:加 PDF 自动识别标识符补全主字段,失败不阻断、可手填/重试/覆盖。(均在客户端做,结果走 Phase 2 同步)

- [ ] 3.1 `IdentifierExtractor`:扫前几页文本(已有 `QPdfDocument`/`BlockClusterer`)取 DOI→arXiv→ISBN→PMID,**复用 Zotero 公开正则**。(M)
- [ ] 3.2 `MetadataResolver`(`QNetworkAccessManager`):DOI→`doi.org` 内容协商**直接拿 CSL-JSON**(存 `items.csljson`,顺带为将来留用);arXiv→`export.arxiv.org` Atom(`QXmlStreamReader`);可选 PMID/ISBN(Open Library)。(L)
- [ ] 3.3 流程:入库后异步补全;失败清晰提示 + 允许手输标识符/手填(不阻断);可重试/覆盖;无网只入库待补全。(M)

**判据**:有 DOI/arXiv 的 PDF 导入后主字段自动填、数秒级;失败可手填且不卡流程。

## Phase 4 · 组织 + 全库检索

**目标**:多层分组 + 多标签(随课题同步);课题内全文检索定位到文献并尽量到段落;字段/标签/分组筛选。

- [ ] 4.1 `collections` / `tags` / `item_tags` 双端 + 纳入同步引擎(对象通用,Phase 2.5 已支持)。(M)
- [ ] 4.2 本地 **FTS5** 虚表:`fts5(item_id UNINDEXED, kind, content)`,正文取自现有 `BlockCache`;入库/同步入库时写索引。(M)
- [ ] 4.3 `SearchService`:`MATCH` + BM25 排序 + `snippet()`;正文命中带 block 序号 → 打开滚到该段(把现有单篇 `runSearchPaper` 升级为**课题全库**)。(M)
- [ ] 4.4 UI:collections 树 + tags 过滤 + 搜索框 + 结果列表(点击打开并定位)。(L)

**说明**:**检索在本地(FTS5),数据靠同步**——既快又离线可用。砍掉 Zotero 手搓倒排那一整套。

**判据**:A 跑的摘要,B 在同课题能看到并标明出自 A。

## 8. 决策点(建议先拍板再进对应 Phase)

1. **后端框架**:NestJS(结构全、适合多资源+权限)〔推荐〕 vs Fastify/Hono(更轻)。
2. **ORM**:Prisma〔推荐〕 vs Drizzle(更轻/SQL-first)。
3. **对象存储**:MinIO 自托管〔推荐,与自建一致〕 vs Cloudflare R2 / S3(省运维但又引入外部依赖)。
4. **AI 摘要共享粒度**(Phase 5.4):每人一份 + 可设课题首选〔倾向〕 vs 课题共享一份可覆盖。
5. **同步冲突策略**:字段级后写胜〔起步推荐,简单〕 vs 引入 CRDT(仅笔记类需要时再上)。
6. **托管位置**:自己的 VPS / 公司服务器?(影响备份、合规、成本)——纯自建必答。

## 9. 风险 / 须正视

- **R1(高)**:**自建后端是第二个产品 + 长期运维**(认证安全、备份、在线率、补丁)。这是本方向最大成本,非一次性。原需求文档已预警「完全自建同步服务端基本不可行」——选了就要配相应人力。
- **R2(高)**:**多写冲突合并**(Phase 2.5)是同步系统的核心难点,需充分测试(并发改同一题录、离线久了再上线、删除与编辑撞车)。
- **R3(中)**:QtQuick.Pdf 高亮叠加(挂起的标注功能)——启动前 spike。
- **R4(中)**:Qt qsqlite 是否编入 FTS5(Phase 4.2),回退见 v1 备选。
- **R5(低)**:外部题录 API(Crossref/arXiv)速率/可用性——超时+重试+缓存。
